OllamaDev Documentation

A local, private, dependency-free AI coding agent for your terminal.

Getting started

OllamaDev is a terminal coding agent that runs against your local Ollama instance — with optional Ollama cloud models when you want frontier horsepower. It's a single self-contained vanilla PHP binary — no Composer, no extensions beyond PHP core + curl — with diff-previewed edits, undo, an interruptible agent loop, real context management, and 60+ tools.

Requirements:

  • PHP 8.0+ (CLI) with curl — that's it.
  • Ollama running locally (ollama serve).
  • At least one model pulled, e.g. ollama pull qwen2.5-coder.
Recommended models for reliable tool use: qwen2.5-coder, qwen3-coder, mistral, codestral, llama3.1, gpt-oss.

CLI · Desktop · Web — what each needs

OllamaDev runs three ways, all over the same local engine (your Ollama models), the same agent, tools, crew, and the same shared ~/.ollamadev data (sessions, memory, crew packs, skills). Pick one or use all three — work flows between them. The desktop and web app share one frontend: an infinite canvas where every tool is a draggable window.

CLI

Zero-dependency standalone binary (PHP is bundled in). Install and run:

curl -fsSL https://raw.githubusercontent.com/kennethyork/OllamaDev/main/install.sh | sh
ollamadev                 # interactive · or: ollamadev crew "…", ollamadev watch "…"

From source instead: PHP 8.0+, then make install (see Installation). Needs only php + curl.

Desktop (ADE)

Native windowed GUI — an infinite canvas of draggable windows (terminals, editor, files, the live crew topology, voice, browser preview, board). Download the archive for your OS, unzip, and run — it bundles the CLI and the Boson libraries:

./OllamaDev-ADE           # Windows: OllamaDev-ADE.bat

Needs PHP 8.4+ (the Boson WebView runtime). From source: composer install then php index.php.

Web

The same ADE in a browser — no native window, no Boson. The archive bundles a launcher:

./OllamaDev-Web           # Windows: OllamaDev-Web.bat → open http://localhost:41434
# from source:  cd Desktop/ollamadev-ade && composer serve

Needs PHP 8.0+ and a browser — web mode doesn't load Boson, so the 8.4 desktop requirement doesn't apply (running it from the prebuilt archive needs no Composer at all). Localhost-only by default; for remote access see Self-hosting on a VPS.

Using it — open http://localhost:41434 and you get the same ADE as the desktop (identical UI; only the bridge differs). The workspace is one infinite, pannable/zoomable canvas — every tool is a draggable window. A quick tour:

  1. Add windows — the ＋ Add button (or right-click the canvas) drops a Terminal, Chat, Editor, Files, Search, Board, Graph, Browser, Voice, Crew topology, Roles, Skills, or Review window where you click. Drag the header to move, the corner to resize, ⤢ to focus.
  2. Open a project — the ☰ drawer holds the rail and the Projects switcher; pick a folder and the whole canvas (windows, positions, session) is restored per project.
  3. Terminals — real PTYs with full-screen TUI support (vim, htop). On a phone, use the touch bar (type a line + Enter, or the Tab/Esc/↑↓/Ctrl-C keys).
  4. Chat — the 💬 Chat window is a plain chatbot (no agent, no tools, no file access): pick any local model from the in-window picker and just talk. It runs ollamadev chat, streams replies, and remembers your last model.
  5. Crew + cockpit — 👥 Crew runs a multi-agent task; the 🛰 Crew topology window shows it live (each coder's branch, the files it's touching, the Auditor's verdict, and ✎/👁/⚡ activity) alongside the Board.
  6. Voice control — the 🎙 Voice window is press-to-talk (bundled Whisper, CPU-only): dictate to a terminal, open windows, or “start a crew to add tests”. Browsers need HTTPS/localhost for the mic — see Self-hosting.

Because it shares ~/.ollamadev with the CLI and desktop, your sessions, memory, crew packs, and skills are the same across all three.

Installation

Download a binary (easiest)

Standalone, PHP-free native binaries — nothing else to install. One-liner for Linux/macOS:

curl -fsSL https://raw.githubusercontent.com/kennethyork/OllamaDev/main/install.sh | sh

Or pick your platform (CLI + desktop app) on the Downloads page. Update in place anytime with ollamadev update --install.

From source

git clone https://github.com/kennethyork/OllamaDev.git
cd OllamaDev

# Build + run the test suite + install to ~/.local/bin/ollamadev
make install

# or just build the single-file binary in place
./build.sh        # produces ./ollamadev

make install only installs if the build and the offline smoke suite pass, and it backs up the previously installed binary so you can make rollback instantly. Make sure ~/.local/bin is on your PATH, then run ollamadev help.

Desktop app (ADE)

OllamaDev ADE (Agentic Development Environment) is an optional desktop GUI in Desktop/ollamadev-ade/ — a Boson WebView app built with vanilla PHP/JS/HTML/CSS (no front-end frameworks). It wraps the agent in a windowed interface with a file browser and a real PTY terminal.

Unlike the CLI binary (which is dependency-free), the desktop app uses Composer for the Boson runtime and needs PHP 8.4+.

Grab a prebuilt desktop archive for your OS from the Downloads page — unzip and run ./OllamaDev-ADE (Windows: OllamaDev-ADE.bat). It bundles the agent CLI and Boson's platform libraries. Or run from source:

cd Desktop/ollamadev-ade
composer install
php index.php        # launch the desktop window (PHP 8.4+)
composer build       # package OS archives → dist/binaries/
Boson runtime ^0.19 is a runtime, not a compiler — there's no single-file boson compile. The desktop ships as a portable app archive (app + Boson libs + bundled CLI + launcher), so it needs PHP 8.4+ to run, unlike the standalone CLI binary.

No-install desktop builds (PHP bundled)

If you'd rather not install PHP 8.4, two self-contained builds bundle their own PHP runtime — download, run, done. Both are on the Downloads page.

🐧 Linux — AppImage (x86_64 and aarch64). One portable file with PHP + the Boson libs + the agent CLI and the local voice (whisper.cpp + model) engine baked in — so /voice works offline too. Nothing to install:

chmod +x OllamaDev-ADE-x86_64.AppImage   # or -aarch64 on ARM
./OllamaDev-ADE-x86_64.AppImage          # double-click also works

Built with scripts/build-appimage.sh (arch-aware: bundles that arch's PHP + matching Boson lib). The arm64 image is built on a native ARM runner. The .tar.gz archive (system PHP) stays as a fallback.

🪟 Windows — installer (OllamaDev-ADE-Setup.exe). A self-contained Inno Setup installer that bundles a PHP 8.4 runtime (with the VC++ runtime + the Boson DLL + the agent CLI), so it installs and runs with no separate PHP install — the Windows equivalent of the AppImage. Double-click to install; it adds Start-menu and optional desktop shortcuts and launches windowless via php-win.exe.

The installer is unsigned, so on first run Windows SmartScreen may show a “Windows protected your PC” warning — click More info → Run anyway. The .zip archive (system PHP) remains as a fallback.

There's no native macOS app — macOS uses the .tar.gz archive (needs system PHP 8.4+), since a signed/notarized .app requires an Apple Developer account.

Why PHP & make install?

OllamaDev is written in vanilla PHP. The ollamadev "binary" is a PHP script with a #!/usr/bin/env php shebang — not a compiled executable — so php is its runtime. There's no compile step: you edit the modules in src/, and build.sh concatenates them into one self-contained file you can read and audit.

  • Why PHP? It's the project's language and ships almost everywhere. The CLI needs only php 8.0+ and curl — no Composer, no npm, no toolchain. (The desktop app needs PHP 8.4+ for its Boson runtime.)
  • Why make? It's just a convenience wrapper around the workflow. make install runs build → test → install in order, and because each step depends on the previous, it only installs when the build lints clean and all tests pass — then backs up the old binary so make rollback is instant.

You can always skip make and do it by hand:

./build.sh                    # concatenate src/ into ./ollamadev
cp ollamadev ~/.local/bin/    # install it yourself

Developing OllamaDev with OllamaDev

You can dogfood it — point the CLI or desktop ADE at this repo and let your local model build features for OllamaDev itself. It's built for that:

  • Self-modification is guarded. When Crew detects it's running on the OllamaDev source (it finds build.sh + the OLLAMADEV_VERSION header), it forces review mode — every branch is held, nothing auto-merges. Interactive edits still show a diff and ask first; every change is checkpointed (/undo reverts).
  • The vanilla rule is enforced at build time — no frameworks, no package.json/CDN, deps limited to PHP + Boson — so the agent can't pull in a dependency.

The loop is edit → review → build → test → verify → commit:

  1. Run the agent in ask mode (diff + approve), or ollamadev crew "<task>" (review is forced on this repo).
  2. Review the diff before it lands.
  3. If src/*.php changed, ./build.sh — the running CLI/ADE won't see src/ edits until you rebuild and restart. Desktop UI edits (public/*) only need a window reload, no build.
  4. php tests/smoke.php (offline, deterministic) — your safety net.
  5. Reload/restart to verify in the real app, then commit when you decide.
Your local model's quality is the ceiling — keep changes scoped, lean on the diff review + smoke tests + /undo, and don't run it unattended on its own source. A larger coder model (--coder-model <model>) helps for subtle work.

Context & input

  • @file mentions — explain @src/foo.php inlines the file into your message.
  • Image / vision input — attach images with @image.png, @~/shot.png, or /image <path> for multimodal models. OllamaDev base64-encodes them into Ollama's images field, and warns (pointing at an installed vision model) if the active model can't see — pull one with ollamadev models pull llava.
  • Voice input (local STT) — nothing to install — /voice records your mic and transcribes it locally, then submits the text as your prompt. The Whisper engine is baked in: the desktop builds (AppImage / Windows installer) bundle a self-contained whisper.cpp + model so it works instantly & offline; the standalone CLI auto-downloads it once on first /voice (into ~/.ollamadev/stt), then runs fully offline forever. CPU-only — nothing leaves the machine. (Prefer to bring your own? Drop a whisper.cpp binary + ggml model into ~/.ollamadev/stt, or pip install -U openai-whisper.) Press Enter to stop. Accuracy: /voice model <tiny|base|small|medium|large-v3|turbo> (small is the CPU sweet spot); history: /voice history; non-interactive install: ollamadev voice install. In the desktop/web app it's the 🎤 mic button plus a voice-model dropdown and a 🎙 History panel in the header.
  • Real token/context meter — /status and /context show actual token counts and context fill.
  • Auto-compaction — older messages are summarized by the model to stay within the window. Fires automatically on real context fill (≥75% of num_ctx, from Ollama's actual prompt-token count) or a message-count threshold, keeping the most recent turns verbatim and preserving tool output the recent steps still reference. Tune with agents.compactContextPct / agents.compactThreshold / agents.compactKeep; force it with /compact.
  • Checkpoint & undo — every file change is snapshotted; /undo reverts the last one, /checkpoints lists them.

Workspaces

A workspace is a named project folder you switch between. The list is global — stored once in ~/.ollamadev/workspaces.json — so a workspace you add in the terminal shows up in the desktop app and the browser, and vice-versa. Because sessions, graph memory, and Crew runs are already keyed by the working directory, opening a workspace automatically resumes that project's session and memory; workspaces add a named switcher and saved window state on top.

Desktop & web

  • Project list — a vertical list at the top of the left pane (under a "Projects" header), one row per workspace. Click a project to switch; the active one is highlighted. The list keeps a stable order so it doesn't jump around.
  • Full state restore — switching tabs opens that project's folder and brings back its terminals, open editor tabs, layout, and active view. Within a session the terminals re-attach to their still-running processes (a build keeps going); after an app restart they respawn fresh.
  • Open / close — ＋ Open project at the bottom of the list (or 📂 Open) opens another folder as a new project; × on a row closes it (removes it from the list — your files are untouched), and the neighbouring project takes over.
  • On launch the app reopens your active workspace instead of prompting for a folder.

Tabs are a fast switcher, not split panes — one project fills the window at a time (the others' terminals keep running in the background). To see two projects on screen at once, open the app (or a browser window) twice.

CLI

ollamadev workspace list                 # show all (* = active)
ollamadev workspace add [path] [name]    # add a folder (defaults to the current dir)
ollamadev workspace remove <name>         # drop it from the list
ollamadev workspace open <name>           # print its path

# jump to a workspace in your shell:
cd $(ollamadev workspace open my-app)

ws is a shorthand alias for workspace.

Crew — multi-agent

Crew is a bench of agents: a Researcher gathers context, a Director plans and decomposes the task, parallel Coders implement in isolated git worktrees, and an Auditor reviews before anything lands.

# run a task
ollamadev crew "add a /health route and tests"

# interactive — configure the team, then prompt the Director live
ollamadev crew

# pick a model per role
ollamadev crew "refactor auth" \
  --director-model qwen2.5-coder --coder-model codestral

# run coders across multiple local hosts, in live tmux panes
ollamadev crew "big refactor" --hosts host-a,host-b --panes
  • Per-subtask roles — the Director tags each subtask with a role from a catalog (built-ins: coder, tester, docs, refactor, security), and each coder runs with that role's persona, optional pinned model, and permission mode. The role shows on each card of the live board.
  • Custom roles — add your own: crew role add <name> "<persona>" [--desc "…"] [--model <m>] [--readonly]; manage with crew role list | show | remove. Or use the 🎭 Manage roles panel in the desktop/web Crew screen. Roles are plain JSON in ~/.ollamadev/crew-roles/, one catalog shared by CLI, desktop, and web.
  • Per-role models — --director-model, --coder-model, --auditor-model, --researcher-model (a role can also pin its own).
  • Parallel coders — --parallel [N] runs coders concurrently on one box (bounded pool; set OLLAMA_NUM_PARALLEL for true concurrent inference), or --hosts a,b,c spreads them across local model servers.
  • Live topology (cockpit) — in the ADE, the 🛰 Crew topology window maps the run live: each coder's branch, the files it's touching, the Auditor's verdict, and real-time activity (✎ editing / 👁 reading / ⚡ running). Steer a coder from the Director box, or --panes for one tmux pane per Coder.
  • Voice-driven — in the ADE, launch and steer the crew by voice: “start a crew to add tests”, “tell coder 2 to focus on edge cases”.
  • Review gating — --review holds every branch for review; nothing auto-merges. Self-repo edits are gated automatically.
  • Team skill packs — Crews are seeded with relevant skills by focus; every by-project-type team also auto-loads its own skill. Disable with --no-skills.

Skills

Skills are reusable instruction packs (SKILL.md folders) the model loads on demand via progressive disclosure — only the catalog of names/descriptions sits in the prompt; the full body is fetched when a skill is invoked. 60+ ship built in: capability skills (payments-money, testing-discipline, auth-security…) that crews load by focus, plus a per-team starter for every crew team. View, create, edit, or delete any of them in the ADE's 🧩 Skills window (customizing a built-in saves an override), or author your own from the CLI and share them.

ollamadev skills                       # list discovered skills
ollamadev skills new my-skill          # scaffold a SKILL.md
ollamadev skills install <dir|git-url|.tar.gz|.zip>
ollamadev skills export my-skill       # package to share
ollamadev skills remove my-skill
A skill is instructions the model may follow. Review what you install — your permission mode (auto/ask/readonly) still gates writes and shell.

Graph memory

A wiki-linked knowledge base that persists across sessions. Notes link to each other with [[slug]]; the agent sees an index in its prompt and recalls full notes on demand. Two tools drive it: recall (read-only) and remember (write).

ollamadev memory                  # list notes
ollamadev memory new "API design"  # create a note
ollamadev memory show <slug>        # read a note
ollamadev memory search <query>     # search notes
ollamadev memory graph             # print the link graph (--json for data)

The desktop ADE renders this graph visually in its Memory tab.

MCP & LSP

MCP — connect Model Context Protocol servers over the standard stdio transport (JSON-RPC 2.0) or an HTTP/SSE fallback. Configure them under mcpServers in your config, or manage at runtime:

ollamadev mcp list
ollamadev mcp add filesystem "npx -y @modelcontextprotocol/server-filesystem /path"

LSP — built-in code intelligence (goto, find_refs, symbols, hover, diagnostics, format) plus an LSP server mode so editors can use OllamaDev for AI completions and diagnostics: ollamadev lsp --port 4389.

Web search

The agent can search the web (the search tool) and read a page (fetch). Run a search directly, too:

ollamadev search "fastify health route" --limit 5
ollamadev search "rust async" --provider searxng    # self-hosted instance
  • Backends — duckduckgo (default, no key), searxng (self-hosted, most local-first — set search.host), or brave (Brave Search API — set search.key / BRAVE_API_KEY). Choose with search.provider in config or --provider per run.
  • Still local-first — only your query leaves the machine; the AI stays on your hardware. No code or prompt is sent to a cloud model.
  • Search-only switch — search.enabled:false (or config set search.enabled false, or the 🔍 Search top-bar button) turns off web search while leaving fetch and remote git working.
  • Web-access toggle — --no-web / config web.enabled:false blocks the agent's network tools (search & fetch & remote git) for a run; local work is untouched. In the desktop & browser, the 🌐 Web button flips the same setting, shared with the CLI.

Code intelligence

Three commands make the agent work like it knows your codebase — all local:

# find code by meaning, not keywords (local embeddings)
ollamadev index build          # embed the repo (model: embed.model, default nomic-embed-text)
ollamadev code-search "where do we gate the agent's network tools"

# test-aware: run the suite, or fix failures until green
ollamadev test                 # auto-detects npm/phpunit/composer/go/cargo/pytest/make
ollamadev verify --max 4       # run → agent fixes failures → re-run, until green

# ship it
ollamadev commit -a            # AI Conventional Commit message from the diff
ollamadev pr create            # push + open a PR (AI title/body) via gh
ollamadev pr review 42 --comment   # review a PR with the local model
  • Semantic search needs an embedding model pulled once: ollama pull nomic-embed-text. The index lives in .ollamadev/index/ and is rebuilt with index build. The agent can call it as the code_search tool, and the desktop/browser have a 🔎 search panel in the sidebar (type a query → click a result to open it).
  • verify and the run_tests tool let the agent confirm its own changes; set test.command to override detection.
  • PR commands use the GitHub CLI (gh); commit is local and always available.

Configuration

Config is read from the first of: ~/.ollamadev/config.json, ~/.config/ollamadev/config.json, .ollamadev.json (project-local). Environment variables override the config file.

{
  "provider": "ollama",
  "ollama": {
    "host": "http://localhost:11434",
    "defaultModel": "qwen3.5:9b",
    "contextWindow": 16384,
    "temperature": 0.3,
    "stream": true
  },
  "model": { "nagWeakModel": true },
  "stt": { "model": "base", "language": "", "maxSeconds": 60, "host": "", "command": "" },
  "agents": {
    "maxIterations": 12,
    "maxToolOutput": 12000,
    "subagentPermission": "readonly"
  },
  "permissions": { "mode": "ask" },
  "web": { "enabled": true },
  "search": { "enabled": true, "provider": "duckduckgo", "host": "", "key": "" },
  "embed": { "model": "nomic-embed-text" },
  "test": { "command": "" },
  "ui": { "markdown": true },
  "hooks": { "afterEdit": "echo edited" },
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path"]
    }
  }
}

Voice (stt): model is the Whisper size (tiny…large-v3/turbo); language empty = auto-detect; maxSeconds caps a recording. Leave host/command empty to auto-detect a local Whisper engine — or set command to drive your own (e.g. "whisper-cli -m model.bin -f {file} -nt") / host for a local transcription server. All local. model.nagWeakModel: set false to silence the startup hint that suggests a stronger installed model for tool use.

Cloud models (optional)

OllamaDev is Ollama-only. Beyond local models, you can opt into Ollama's hosted cloud models — frontier-scale models that run on Ollama's servers but are reached through your local Ollama daemon, so it's still one backend and the same -m <tag> everywhere (CLI, desktop, web, and the crew — mix local + cloud per role).

ollama signin                          # once — creates a free ollama.com key
ollamadev models cloud                 # the curated catalog + how to enable
ollamadev models pull qwen3-coder-cloud # or any :cloud / -cloud tag
ollamadev -m qwen3-coder:480b-cloud    # use it (also /model in chat)
Cloud models leave the machine — prompts run on Ollama's servers (the catalog says so plainly). Local stays the default; cloud is strictly opt-in. To keep a run local, the 🌐 Web toggle / --no-web blocks the agent's network tools.

Cloud models always run at their full context window — the local maxContextWindow cap (which exists to fit your GPU/RAM) is bypassed for them, since they run on Ollama's hardware. E.g. a 512K-context cloud model uses all 512K.

How the cloud login works

Cloud access lives in Ollama, not OllamaDev — OllamaDev never sees an API key, it just relays through your local daemon. You enable cloud by running ollama signin (a free ollama.com account) once on the machine that runs Ollama; after that every :cloud model just works. There is no per-user login inside OllamaDev, which means it behaves differently per surface:

  • CLI & desktop — each person signs in on their own machine, so cloud is genuinely per-user: their account, their usage, their billing.
  • Self-hosted web (VPS) — there's one Ollama daemon, so every browser user shares the single account you ran ollama signin with on the server. Same cloud identity, same usage limits and billing for everyone — there's no per-visitor cloud auth.
Quota and cost are governed by that ollama.com account, not by OllamaDev. Local models need no login at all — ollama signin only matters for the optional :cloud models. If you expose the web app to others and don't want them spending your cloud quota, stick to local models on that server (or keep the :cloud tags unpulled).

Self-hosting on a VPS

Run OllamaDev on a server you control and reach it from anywhere — handy when you're away from your desktop. Everything still runs on your box (a VPS is your server, not a cloud AI provider), so it stays self-hosted.

Heads-up if more than one person will use it: there's no per-user login, and any Ollama cloud models on the server run under the single account you ollama signin'd there — everyone shares that cloud identity and its billing. See How the cloud login works above.

Two ways to use it remotely

Both run the same agent on the VPS — they differ only in the front-end you drive it with. You don't need both; pick what fits the moment.

PathWhat you doBest for
Plain SSH + the CLI
no web server, no tunnel, no token		 ssh you@your-vps then ollamadev A terminal you're already in (laptop). The full agent — diffs, undo, Crew, memory — in the shell.
SSH tunnel + web mode
the browser GUI		 ssh -L 41434:localhost:41434 you@your-vps, then open http://localhost:41434/?token=… A graphical interface on a phone/tablet — file tree, editor, Crew board, memory graph, touch terminals, voice.
The browser interface is web mode — the SSH tunnel is just the secure pipe that carries it, not a separate version. If the terminal CLI over SSH is enough for you, you don't need to run the web server at all. The rest of this page sets up web mode (the GUI path); the CLI path needs nothing beyond ollamadev on the VPS.

One-shot setup script

Installs Ollama + a model, fetches the ADE, and starts web mode bound to localhost with a randomly-generated auth token — so it is not reachable from the internet and even other users on the box can't use it without the token. Then reach it over an SSH tunnel:

# PHP 8.0+ must already be installed (e.g. sudo apt install -y php-cli)
curl -fsSL https://raw.githubusercontent.com/kennethyork/OllamaDev/main/scripts/vps-setup.sh | bash
# override: MODEL=llama3.2:3b PORT=41434 CLI=0 FIREWALL=1 … | bash
It picks a small-model default, sets up an always-on systemd service when it can, generates an OLLAMADEV_SERVE_TOKEN, and prints your token plus the exact ssh -L tunnel command and http://localhost:PORT/?token=… URL at the end. Pass FIREWALL=1 to also enable ufw (it allows SSH first, then keeps the app port private). Set OLLAMADEV_SERVE_TOKEN yourself to pin a specific token. Prefer to do it by hand? The manual steps follow.

1. Install Ollama + a model

curl -fsSL https://ollama.com/install.sh | sh   # installs + a systemd service
systemctl status ollama                          # runs on localhost:11434
ollama pull qwen2.5-coder:7b                      # or a smaller model — see the note
Most VPSes have no GPU, so inference is CPU-bound (slower) and model size is capped by RAM. Prefer small models (llama3.2:3b, qwen2.5-coder:3b/7b). Run ollamadev context on the VPS for a num_ctx that fits its memory. Expect lower quality than a local GPU box.

2. Get the web app + run it

On a headless VPS, download and extract the ADE archive (it bundles the agent CLI and needs no Boson), then start the browser app. Pick linux-x64 or linux-arm64 to match your VPS:

# download + extract the ADE (one line, no GUI needed)
curl -fsSL https://github.com/kennethyork/OllamaDev/releases/latest/download/OllamaDev-ADE-linux-x64.tar.gz | tar -xz
cd OllamaDev-ADE
./OllamaDev-Web            # → http://localhost:41434  (needs PHP 8.0+)

That's all web mode needs — the archive ships its own bin/ollamadev. If you also want the ollamadev command in your shell (for crew/watch over SSH), add the standalone CLI too:

curl -fsSL https://raw.githubusercontent.com/kennethyork/OllamaDev/main/install.sh | sh

3. Reach it securely from anywhere

The web app exposes shell, PTY, and file-write over HTTP — never put it on the open internet raw. Two safe options:

Best — SSH tunnel (keep it localhost on the VPS, encrypt the hop):

# on your laptop:
ssh -L 41434:localhost:41434 you@your-vps
# then open http://localhost:41434/?token=… — traffic is tunnelled, nothing public

With this setup the app is on 127.0.0.1 only, so the sole way in from outside is forwarding the port over SSH — which means only someone who can SSH into that account can reach the ADE, and access lasts exactly as long as the tunnel is open. To make that truly just you, lock SSH to your key:

# on the VPS — key-only SSH, no passwords:
sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart ssh

The OLLAMADEV_SERVE_TOKEN is then a second lock — even another user already on the box can't use the API without it. (Tailscale gives the same guarantee for phones without an SSH client: the port is reachable only from your private tailnet.)

Or bind public with a token (and firewall the port to your IP):

export OLLAMADEV_SERVE_TOKEN="$(openssl rand -hex 16)"
OLLAMADEV_SERVE_HOST=0.0.0.0 ./OllamaDev-Web
# open http://your-vps:41434/?token=<the token>
Keep Ollama's port 11434 private (localhost). Only the web app port is exposed, and only behind a tunnel or token + firewall. For sensitive code, prefer your local desktop — a VPS is yours, but it's off your machine.

HTTPS & the microphone

Ollama serves plain HTTP and has no built-in TLS — but it doesn't need it here: OllamaDev talks to Ollama over localhost on the VPS, which never leaves the box. The hop that crosses the internet is your browser → the web app, and that's what to encrypt:

  • SSH tunnel (simplest) — encrypts the hop and makes the app localhost on your device; no certificates.
  • HTTPS via a reverse proxy — front the web app with Caddy (automatic Let's Encrypt) or nginx, and pair it with OLLAMADEV_SERVE_TOKEN:
# Caddyfile — terminates HTTPS, proxies to the local web app
ade.example.com {
    reverse_proxy localhost:41434
}
The mic needs a secure context. Browsers only allow the microphone (the voice-input feature) over HTTPS or localhost. So voice works over an SSH tunnel (it's localhost) or behind HTTPS — but on plain http://your-vps:41434 the browser silently blocks it. For remote voice, use the tunnel or real HTTPS.

Using it from your phone

The ADE is responsive, so once the VPS is reachable you can drive it from a phone browser — file tree, crew board, memory graph, terminals (with the touch key bar), even voice. No control panel needed.

First, keep the server running so it survives your SSH session ending — a tiny systemd service:

# /etc/systemd/system/ollamadev-web.service
[Unit]
Description=OllamaDev ADE (web)
After=network.target
[Service]
WorkingDirectory=/home/you/OllamaDev-ADE
ExecStart=/home/you/OllamaDev-ADE/OllamaDev-Web
Restart=on-failure
User=you
[Install]
WantedBy=multi-user.target
sudo systemctl enable --now ollamadev-web   # now always serving localhost:41434 on the VPS

Option A — SSH tunnel from the phone (simple; voice works because it's localhost):

  • Install a phone SSH app that supports port forwarding — Termius (iOS/Android), Blink (iOS), or JuiceSSH (Android).
  • Add a local port forward: phone localhost:41434 → VPS localhost:41434, then connect.
  • Open http://localhost:41434 in the phone's browser. That's it.

Option B — Tailscale (easiest on a phone, and gives HTTPS so the mic works without an SSH app):

# on the VPS
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
sudo tailscale serve --bg 41434     # serves localhost:41434 as HTTPS on your tailnet
  • Install the Tailscale app on your phone and sign in with the same account.
  • Open the https://<host>.<tailnet>.ts.net URL that serve prints — done. No SSH, no certificates, and voice works (it's HTTPS).
Both ways keep Ollama (11434) and the app (41434) bound to localhost on the VPS — nothing is exposed to the public internet. Your phone reaches them only through the encrypted SSH tunnel or your private tailnet.

Troubleshooting

Cannot connect to Ollama — start it with ollama serve, or set OLLAMA_HOST. (On launch, OllamaDev prints the exact fix-up steps for whichever backend is configured.)

Model not found / no models installed — ollama pull qwen2.5-coder, or just /pull qwen2.5-coder in-session.

Unreliable tool use — small models struggle with tool-calling; try a recommended model (qwen2.5-coder, mistral, codestral) or use /chat for plain conversation.

Permission denied running the binary — chmod +x ~/.local/bin/ollamadev.

Need to undo a change — /undo reverts the last edit; /checkpoints lists every snapshot. Rolling back an install: make rollback.
